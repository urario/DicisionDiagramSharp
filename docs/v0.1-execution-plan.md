# v0.1 Execution Plan: ZDD Foundation

This plan translates `AGENTS.md`, `docs/architecture.md`, `docs/backlog.md`, and
`docs/done-policy.md` into an executable v0.1 delivery sequence.

v0.1 focuses on a verified ZDD foundation. BDD, CodeAnalysis, CLI, and unified
high-level developer experience are intentionally deferred unless they are needed
as scaffolding for v0.1 artifacts.

> Status note (2026-05-07): v0.1 implementation and verification are completed.
> Execution evidence is tracked in `docs/backlog.md` under "v0.1 Execution Evidence".

## 2. Delivery Strategy

v0.1 is split into two gates:

1. **ZDD Core MVP**
   - Core project, tests, and ZDD semantics are complete.
   - This gate proves the mathematical model and canonical representation.
2. **v0.1 Release Candidate**
   - Diagnostics, export, samples, docs, benchmarks, and evidence are complete.
   - This gate satisfies the full v0.1 milestone definition in `AGENTS.md`.

This avoids letting text export or samples drive the design before the ZDD
semantics are correct.

## 6. API Shape Notes

### 6.1 Core

Initial low-level API should be manager-oriented:

```csharp
var manager = new ZddManager();
var a = manager.GetOrAddVariable("A");
var family = manager.MakeFamily(new[]
{
    new[] { a }
});
```

Avoid making v0.1 depend on the future `DecisionDiagramManager` unless the
implementation cost is small and it does not blur the v0.3 milestone.

### 6.2 Diagnostics and Export

Prefer this flow:

```text
Zdd -> Diagnostics model -> TableModel -> CSV / Markdown / AsciiDoc
Zdd -> DOT diagnostics string
```

DOT belongs to Diagnostics. CSV, Markdown, and AsciiDoc belong to Export.

### 6.3 Friendly API

User-friendly fluent APIs can be introduced in v0.1 samples only if they are
thin wrappers over the manager API. Full unified developer experience remains
v0.3 scope.

- `Change`

Test dimensions:

- variable counts from 0 to 6
- family sizes from 0 to 32

Golden files should cover:

- DOT for terminals
- DOT for a small non-terminal ZDD

## 9. Risks and Decisions

| Item | Risk / Decision | Plan |
|---|---|---|
| License mismatch | `architecture.md` mentions MIT in one place, while `AGENTS.md` and csproj example recommend Apache-2.0 | Use Apache-2.0 for v0.1 unless project owner decides otherwise; update docs consistently |
| v0.1 project split | `architecture.md` says minimal v0.1 may start with Core only, while `AGENTS.md` requires diagnostics/export in v0.1 | Use Core MVP as first gate, then add Diagnostics/Export before v0.1 RC |
| `DecisionDiagramManager` timing | Architecture roadmap places unified manager in v0.3 | Do not require it for v0.1; keep ZDD manager API stable enough for v0.3 wrappers |
| Operation semantics | `Subset1` and `Containing` are easy to conflate | Require separate naive comparison tests and documentation examples |
| Enumeration explosion | ZDD set enumeration can be large | All public enumeration diagnostics must be bounded by default |
| Over-abstraction | BDD and ZDD internals may be prematurely unified | Commonize only variable IDs, variable table, statistics, diagnostics concepts, and validation conventions |
