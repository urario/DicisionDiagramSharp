# v0.7.1-v0.7.5 Test Quality Remediation Plan

## Scope

This plan turns the findings in `docs/reviews/unit-test-review.md` into executable backlog work.

The work is intentionally limited to test quality, test organization, and documentation consistency. It does not add new diagram semantics and does not change production behavior unless a later implementation task discovers a genuine product bug while strengthening tests.

## Source Review

Primary input:

- `docs/reviews/unit-test-review.md`

The review found that the current test suite has useful coverage and several strong oracles, but also has serious maintainability risks:

- white-box reflection tests are mixed with public API specification tests
- some tests assert invalid hash-code properties
- several BDD algebra tests use BDD operations as their own expected oracle
- some test names do not match the asserted behavior
- large test classes mix unrelated specification categories
- randomized tests are useful but shallow in places

## Version Goals

| Version | Goal | Main outcome |
|---|---|---|
| v0.7.1 | Correct misleading tests and invalid assertions | Test names and assertions no longer lie about the specification |
| v0.7.2 | Isolate white-box tests | Reflection-based tests are categorized and separated from public API tests |
| v0.7.3 | Strengthen BDD semantic oracles | BDD Boolean semantics are verified by explicit truth tables or independent evaluators |
| v0.7.4 | Split large test classes | BDD tests are organized by specification area |
| v0.7.5 | Strengthen randomized/property-like tests | Randomized tests use reproducible compound structures and independent oracles |

## Non-Goals

- Do not change production code as part of planning.
- Do not delete reflection tests before they are isolated and reviewed.
- Do not lower coverage or quality gates to make the work easier.
- Do not mark a task `Done` only because coverage is high.
- Do not merge semantic tests and canonicalization tests when their purposes differ.

## Execution Order

```text
v0.7.1
  -> v0.7.2
      -> v0.7.3
          -> v0.7.4
              -> v0.7.5
```

The order matters. Rename and assertion fixes should land before moving files. White-box isolation should happen before deeper semantic rewrites, so public API test quality can be judged without reflection noise.

## Strict Definition of Done

These rules apply to every v0.7.1-v0.7.5 task:

- The task must update `docs/backlog.md` evidence before being marked `Done`.
- `dotnet test DecisionDiagramSharp.slnx` must pass after every task that changes test code.
- If test code changes production coverage measurement, coverage must be recorded; if no production code changes, coverage target is `N/A; no production code changed`, but the full test pass is still mandatory.
- No task may be marked `Done` if any renamed test name, XML summary, XML remarks, and Assert body disagree.
- Reflection-based tests must either be isolated with `TestCategory("WhiteBox")`, `TestCategory("InternalInvariant")`, and `TestCategory("Fragile")`, or explicitly listed as unresolved follow-up.
- Hash-code tests may assert equal values have equal hash codes. They must not assert unequal values have unequal hash codes.
- BDD semantic tests must not compute expected results with the same BDD operations under test.
- Randomized tests must have a fixed seed and failure messages that include enough information to reproduce the failing case.

The following consistency decisions apply across docs:

| Topic | Decision |
|---|---|

| White-box tests | They may exist, but only when isolated, categorized, and documented as fragile internal invariant tests. |
| BDD oracle policy | Semantic tests require explicit truth tables or independent evaluators. BDD-operation-built expectations are allowed only for clearly labeled canonicalization/algebra identity tests. |
